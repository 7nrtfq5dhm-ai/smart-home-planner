Use Cases

UC-01: Регистрация нового пользователя​

ПолеЗначение
IDUC-01
АкторыНеавторизованный пользователь, Email/SMS Service, Backend API
ПредусловияПользователь открыл сайт, не авторизован, есть доступ к email или телефону
Постусловия (успех)Аккаунт создан, пользователь авторизован, статус подписки = Free
Постусловия (неудача)Аккаунт не создаётся, отображается сообщение об ошибке

Основной поток​

  1. Пользователь нажимает «Зарегистрироваться»
  2. Система отображает форму ввода email/телефона
  3. Пользователь вводит email или номер телефона
  4. Система валидирует формат данных
  5. Система отправляет OTP-код подтверждения
  6. Пользователь вводит полученный код
  7. Система проверяет корректность кода
  8. Система открывает форму создания пароля
  9. Пользователь вводит и повторяет пароль
  10. Система проверяет совпадение паролей
  11. Backend создаёт запись пользователя в БД
  12. Система автоматически авторизует пользователя
  13. Пользователь перенаправляется на главный экран

Альтернативные потоки​

Неверный код подтверждения (вход на шаге 7):

  • Код некорректный или истёк → сообщение «Неверный или истёкший код»
  • Пользователь может запросить повторную отправку

Пароли не совпадают (вход на шаге 10):

  • Сообщение «Пароли не совпадают» → повторный ввод

Email/телефон уже зарегистрирован (вход на шаге 4):

  • Сообщение «Аккаунт уже существует» → Use Case завершается

Бизнес-правила​

  • BR-01: Email или телефон должны быть уникальными
  • BR-02: OTP-код действует 5 минут
  • BR-03: Минимальная длина пароля — 8 символов

UC-02: Авторизация пользователя​

ПолеЗначение
IDUC-02
АкторыЗарегистрированный пользователь, Backend API
Постусловия (успех)Создана активная сессия (JWT), пользователь на главном экране

Основной поток​

  1. Пользователь вводит email/телефон и пароль
  2. Система валидирует формат данных
  3. Backend проверяет корректность учётных данных
  4. При успешной проверке создаётся JWT-токен
  5. Пользователь перенаправляется на главный экран

Бизнес-правила​

  • BR-01: После 5 неудачных попыток вход блокируется на 15 минут

UC-03: Создание нового проекта с нуля​

ПолеЗначение
IDUC-03
АкторыАвторизованный пользователь, Backend API
ПредусловияПользователь авторизован; лимит проектов не исчерпан

Основной поток​

№Действие пользователяОтклик системы
1Нажимает «Создать проект с нуля»Модальное окно с полями: название, ширина × длина
2Вводит название и размеры—
3Нажимает «Создать»Валидация по BR-01, BR-02
4—Backend создаёт запись проекта в БД
5—Редактор открывается с прямоугольником стен

Бизнес-правила​

  • BR-01: Габариты > 0
  • BR-02: Максимальный размер стороны — 100 м
  • BR-03: Для Free-аккаунта — не более 3 одновременных проектов

UC-04: Редактирование планировки через AI-команду​

ПолеЗначение
IDUC-04
АкторыАвторизованный пользователь, Backend API (AI-сервис)

Основной поток​

  1. Пользователь вводит текстовую команду (например: «Сделай кухню больше»)
  2. Нажимает «Отправить»
  3. Фронтенд отправляет команду + состояние проекта на Backend
  4. AI-модуль интерпретирует команду, генерирует изменения
  5. Backend возвращает обновлённые параметры
  6. Редактор перерисовывает план
  7. Уведомление: «Изменения применены»

Бизнес-правила​

  • BR-01: Для Free-подписки счётчик AI-запросов сбрасывается в 00:00 UTC
  • BR-02: AI-команды не могут изменять инженерные коммуникации в MVP
