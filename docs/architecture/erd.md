# ERD — Модель данных

## Основные таблицы и связи

### user → project (1:N)

Один пользователь создаёт много проектов. Внешний ключ `user_id` хранится в таблице `project`.

Все ID используют тип **UUID** вместо обычных числовых идентификаторов — это исключает возможность перебора чужих ресурсов.

### project ↔ asset через project_asset (M:N)

Один объект из каталога (стул, диван) может использоваться в множестве проектов. В одном проекте может быть множество объектов. Промежуточная таблица `project_asset` хранит не только связь, но и атрибуты расположения: координаты `pos_x`, `pos_y` и слой (мебель, электрика и т.д.).

### asset_category — иерархия каталога (петля 1:N)

Каталог объектов организован в дерево категорий. Поле `parent_id` ссылается на ту же таблицу — это позволяет создавать любую глубину вложенности без изменения схемы БД.

### project → ai_task (1:N)

Один проект может содержать множество AI-запросов. История хранится для очереди RabbitMQ и аналитики.

### project → project_canvas_data (1:1)

Метаданные проекта (название, дата, автор) хранятся в `project` — лёгкая таблица для быстрой загрузки дашборда. Сам чертёж в формате JSONB вынесен в отдельную таблицу `project_canvas_data`, чтобы не замедлять запросы к списку проектов.

## Выбор типов данных

| Поле | Тип | Обоснование |
|------|-----|-------------|
| Все ID | UUID | Нельзя угадать/перебрать чужой ресурс |
| Данные холста | JSONB | Гибкая структура, индексируется PostgreSQL |
| Координаты | FLOAT | Точность позиционирования объектов |
| Статусы | VARCHAR / ENUM | Читаемость, ограниченное множество значений |
