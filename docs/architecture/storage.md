# Технологии хранения данных

## Выбор технологий

| Данные | Технология | Обоснование |
|--------|-----------|-------------|
| Пользователи и метаданные проектов | **PostgreSQL** | ACID, сложные связи (FK), строгая целостность |
| Данные чертежа (холст) | **PostgreSQL JSONB** | Гибкая структура плана, поддержка индексов |
| Статусы AI-задач | **Redis** | Молниеносный доступ, не нужно долгое хранение |
| Файлы моделей (2D/3D текстуры) | **S3 Object Storage** | БД не подходят для больших бинарных файлов |
| Аналитика событий | **Data Lake** | Петабайты данных, колоночное хранение для OLAP |

## Паттерн разделения «горячих» и «холодных» данных

При загрузке дашборда система запрашивает только метаданные проектов — название, дату, статус. Сами данные холста (которые могут весить несколько мегабайт) хранятся в отдельной таблице `project_canvas_data` и загружаются только при открытии редактора.

## Интерфейс экранов и API

| Экран | Действие | Endpoint |
|-------|---------|---------|
| Дашборд | Список проектов | `GET /api/v1/projects` |
| Генерация | Создать через AI | `POST /api/v1/projects/generate` |
| Редактор | Загрузить план | `GET /api/v1/projects/{projectId}` |
| Редактор | AI-команда | `POST /api/v1/projects/{projectId}/ai-commands` |
| Редактор | Сохранить | `PUT /api/v1/projects/{projectId}` |
| Редактор | Инженерный слой | `GET /api/v1/projects/{projectId}/layers?type=ELECTRICITY` |
