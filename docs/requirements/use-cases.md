# Use Cases

## UC-01 — Регистрация нового пользователя

| Поле | Значение |
|------|---------|
| **Акторы** | Неавторизованный пользователь, Email/SMS-сервис, Backend API |
| **Предусловия** | Пользователь открыл сайт, не авторизован, есть доступ к email или телефону |
| **Постусловия (успех)** | Аккаунт создан, пользователь авторизован, статус подписки = Free |
| **Постусловия (неудача)** | Аккаунт не создаётся, отображается сообщение об ошибке |

### Основной поток

| № | Действие пользователя | Отклик системы |
|---|----------------------|----------------|
| 1 | Нажимает «Зарегистрироваться» | Форма ввода email или телефона |
| 2 | Вводит email или телефон | Валидация формата данных |
| 3 | — | Отправка OTP-кода подтверждения |
| 4 | Вводит полученный OTP-код | Проверка корректности кода |
| 5 | Вводит и повторяет пароль | Проверка совпадения паролей |
| 6 | — | Backend создаёт запись в БД, выдаёт JWT |
| 7 | — | Редирект на главный экран |

### Альтернативные потоки

**Неверный OTP-код** (шаг 4): сообщение «Неверный или истёкший код», возможность запросить повторно.

**Пароли не совпадают** (шаг 5): сообщение «Пароли не совпадают», повторный ввод.

**Email уже зарегистрирован** (шаг 2): сообщение «Аккаунт уже существует», Use Case завершается.

### Бизнес-правила
- **BR-01** Email или телефон должны быть уникальными
- **BR-02** OTP-код действует 5 минут
- **BR-03** Минимальная длина пароля — 8 символов

## UC-02 — Авторизация пользователя

| Поле | Значение |
|------|---------|
| **Акторы** | Зарегистрированный пользователь, Backend API |
| **Постусловия (успех)** | Создана активная сессия (JWT), пользователь на главном экране |
| **Постусловия (неудача)** | Сессия не создаётся, отображается ошибка |

### Основной поток
1. Пользователь вводит email/телефон и пароль
2. Система валидирует формат данных
3. Backend проверяет учётные данные
4. При успехе создаётся JWT-токен
5. Редирект на главный экран

### Бизнес-правила
- **BR-01** После 5 неудачных попыток — блокировка на 15 минут

## UC-03 — Создание проекта с нуля

| Поле | Значение |
|------|---------|
| **Предусловия** | Пользователь авторизован, лимит проектов не исчерпан |
| **Постусловия (успех)** | Проект создан в БД, открыт редактор |

### Основной поток

| № | Действие | Отклик системы |
|---|---------|----------------|
| 1 | Нажимает «Создать проект» | Модальное окно: название, ширина × длина |
| 2 | Вводит параметры и нажимает «Создать» | Валидация по BR-01, BR-02 |
| 3 | — | Backend создаёт запись, открывается редактор |

### Бизнес-правила
- **BR-01** Габариты > 0 м и ≤ 100 м
- **BR-02** Для Free-аккаунта — не более 3 активных проектов

## UC-04 — Редактирование планировки через AI-команду

| Поле | Значение |
|------|---------|
| **Предусловия** | Проект открыт в редакторе |
| **Постусловия (успех)** | Изменения применены, холст перерисован |

### Основной поток
1. Пользователь вводит текстовую команду («Сделай кухню больше»)
2. Фронтенд отправляет команду + состояние проекта на Backend
3. AI-модуль интерпретирует команду, генерирует изменения
4. Backend возвращает обновлённые параметры
5. Редактор перерисовывает план
6. Уведомление «Изменения применены»

### Бизнес-правила
- **BR-01** Для Free-подписки счётчик AI-запросов сбрасывается в 00:00 UTC
- **BR-02** В MVP AI-команды не изменяют инженерные коммуникации
