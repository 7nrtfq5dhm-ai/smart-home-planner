# Функциональные и нефункциональные требования

:::note Шаблон
Данный раздел содержит шаблон для описания требований. Детальное заполнение — следующий этап работы над документацией.
:::

## Функциональные требования (FR)

Функциональные требования описывают **что система должна делать**.

### FR-01: Управление пользователями
| ID | Требование |
|----|-----------|
| FR-01.1 | Система должна позволять регистрацию через email или номер телефона с OTP-подтверждением |
| FR-01.2 | Система должна поддерживать авторизацию через JWT-токены |
| FR-01.3 | Система должна блокировать вход на 15 минут после 5 неудачных попыток |

### FR-02: Управление проектами
| ID | Требование |
|----|-----------|
| FR-02.1 | Система должна позволять создавать проект с нуля с заданием габаритов помещения |
| FR-02.2 | Система должна позволять генерацию проекта через текстовый запрос к AI |
| FR-02.3 | Система должна ограничивать количество проектов для Free-аккаунта (не более 3) |
| FR-02.4 | Система должна поддерживать автосохранение проекта |

### FR-03: Редактор
| ID | Требование |
|----|-----------|
| FR-03.1 | Редактор должен отображать план помещения на интерактивном холсте |
| FR-03.2 | Редактор должен поддерживать работу со слоями (мебель, электрика, водоснабжение, канализация) |
| FR-03.3 | Редактор должен позволять расстановку объектов из каталога |

### FR-04: AI-помощник
| ID | Требование |
|----|-----------|
| FR-04.1 | Система должна принимать текстовые команды для изменения планировки |
| FR-04.2 | Для Free-аккаунта счётчик AI-запросов должен сбрасываться ежедневно в 00:00 UTC |
| FR-04.3 | В MVP AI-команды не изменяют инженерные коммуникации |

## Нефункциональные требования (NFR)

Нефункциональные требования описывают **как система должна работать**.

### Производительность
| ID | Требование |
|----|-----------|
| NFR-01.1 | Дашборд со списком проектов должен загружаться не более чем за 1 секунду |
| NFR-01.2 | REST API должен отвечать на 95% запросов не более чем за 300 мс |
| NFR-01.3 | _[Указать требования к времени генерации AI]_ |

### Надёжность
| ID | Требование |
|----|-----------|
| NFR-02.1 | Потеря задачи на генерацию AI недопустима: RabbitMQ должен повторить задачу при падении сервиса |
| NFR-02.2 | _[Указать SLA/uptime]_ |

### Безопасность
| ID | Требование |
|----|-----------|
| NFR-03.1 | Все ID ресурсов должны быть UUID для предотвращения перебора |
| NFR-03.2 | Платёжные данные пользователей не хранятся на серверах системы (обработка через ЮKassa) |
| NFR-03.3 | _[Указать требования к шифрованию]_ |

### Масштабируемость
| ID | Требование |
|----|-----------|
| NFR-04.1 | AI-микросервис должен горизонтально масштабироваться за счёт добавления Consumer-воркеров |
| NFR-04.2 | _[Указать ожидаемую нагрузку]_ |
